## Problem Statement

The progress annotator (`demo/progress_annotator.py`) is currently single-user. You want
to collaborate with another researcher: both of you annotate the same MAS failure traces
independently, then compute inter-annotator agreement (IAA) to validate annotation quality
and surface disagreements worth discussing.

The tool needs to be shareable — the collaborator shouldn't need access to the full
mas_error_analysis research codebase (models, evaluators, experiment scripts). They just
need the traces, the annotation UI, and a clear place to save their annotations.

## What Makes This Cool

IAA on structured annotation (plans + root cause attribution) is non-trivial. Most
annotation tools compute agreement on simple categorical labels. Here, agreement requires
matching on:
- Which step in the extracted plan is the root cause (ordinal: step N out of M)
- Per-step annotations (categorical + free text)
- Overall root cause reasoning (free text — harder to compare automatically)

The "whoa" moment: `python compute_iaa.py` → a markdown report that shows per-trace
agreement, overall Cohen's kappa on root_cause_step, a confusion matrix, and flagged
high-disagreement traces worth re-examining together. Paper-ready output.

## Constraints

- Collaborator runs the annotation tool locally (Flask + Python). No hosted server.
- New dedicated git repository — collaborator gets traces + annotation tool without
  access to the full research codebase.
- Schema must remain backward-compatible with existing annotations in mas_error_analysis.
- IAA script should be one command: `python compute_iaa.py`.
- Output should be usable directly in a paper methods section.

## Premises

1. Two annotators, annotating the same traces independently (no real-time collab needed).
2. New dedicated annotation repo — separates the annotation workflow from the research code.
3. No IAA script exists yet — must be built.
4. Schema is stable — `finalized_plan`, `root_cause_step`, `root_cause_reasoning`,
   `step_annotations` are the right fields to compute agreement on.
5. Collaborator runs the Flask tool locally from the new repo.

### Approach C: Complete — Namespaced Dirs + IAA Script + Agreement Report ← chosen
  Summary: Approach B plus: `compute_iaa.py` outputs a markdown report with per-trace

           attribution, and a list of high-disagreement traces.
  Effort:  M
  Risk:    Low
  Pros:    Paper-ready artifact; surfaces disagreements; re-runnable
  Cons:    More upfront code; agreement definition for plan steps needs a decision
  Completeness: 9/10

## Recommended Approach

**Approach C.** The extra work over B is ~100 lines of Python (pandas + sklearn). The
payoff: a report you can include in a paper without manual post-processing. And the
confusion matrix on root_cause_step attribution is the kind of artifact reviewers ask for.

### Config Change to progress_annotator.py

Add `annotator_id` to `config.yaml` (YAML is the source of truth — no Python class change needed for the collaborator, no code editing):

```yaml
# config.yaml (collaborator sets this to "collaborator" before running)

```

In `progress_annotator.py`, read it at startup:

```python
import yaml
with open(PROJECT_ROOT / "config.yaml") as f:
    _cfg = yaml.safe_load(f)

save_dir = PROJECT_ROOT / "data" / "annotations" / annotator_id
```

The collaborator edits `config.yaml`, sets `annotator_id: collaborator`, and runs the tool. No Python edits required.

### compute_iaa.py — Core Logic

```
Usage: python compute_iaa.py [--annotator1 NAME] [--annotator2 NAME]
       Defaults: discovers all subdirs under data/annotations/ and pairs
                 the first two alphabetically. Use flags to override.

Logic:
  1. Load all annotation JSON files from data/annotations/<annotator1>/
     and data/annotations/<annotator2>/.
  2. Find trace_ids present in BOTH sets (intersection).
     Drop excluded=true traces from both sides.
     If overlap < 2: print error and exit.
     If one annotator has >2x the completions of the other: print a warning
     in the report ("annotator A: 200 traces, annotator B: 20 traces — only
     N overlapping traces compared").
  3. For each overlapping trace_id:
     - root_cause_step: record (step_A, step_B) as a label pair.
     - step_annotations: descoped — shown side-by-side in per-trace table only.
  4. Compute metrics on root_cause_step pairs:
     - Percent exact agreement
     - Cohen's kappa (sklearn.metrics.cohen_kappa_score, unweighted)
     - Mean absolute distance |step_A - step_B|
     - Confusion matrix (step indices 1–8; any index >8 labeled "8+",
       included in that bucket, not dropped)
     - Per-trace flag: "agree" (diff=0), "1-off" (diff=1), "disagree" (diff>1)
  5. Write iaa_report.md.
```

`step_annotations` comparison is intentionally descoped: the status vocabulary
is not yet standardized across annotators (see Open Questions #2). The per-trace
table shows both annotators' step_annotations for manual inspection only.

## Annotation Schema (stable)

Each annotation file is a JSON object. Example:

```json
{
  "trace_id": "abc123",
  "finalized_plan": [
    {"step_number": 1, "description": "Search for articles", "depends_on": []},
    {"step_number": 2, "description": "Extract count", "depends_on": [1]},
    {"step_number": 3, "description": "Compute total", "depends_on": [2]}
  ],
  "root_cause_step": 2,
  "root_cause_span_id": "span_xyz",
  "root_cause_reasoning": "The agent failed at extraction step — returned empty result.",
  "step_annotations": [
    {"step_number": 1, "status": "completed", "notes": ""},
    {"step_number": 2, "status": "failed", "notes": "Tool call returned empty"},
    {"step_number": 3, "status": "skipped", "notes": ""}
  ],
  "notes": "",
  "excluded": false
}
```

`root_cause_step` is a 1-indexed integer corresponding to `step_number` in
`finalized_plan`. It can range from 1 to len(finalized_plan). Different traces
have different plan lengths — the IAA script handles this (see Decisions Made).

`root_cause_span_id` is UI metadata only (links the root cause to a span in the
trace tree). It is not used in IAA computation.

`excluded` is a top-level boolean. `true` means the annotator flagged this trace
as unsuitable for the dataset. Excluded traces are dropped from IAA computation.

## Decisions Made

1. **Agreement definition for plan steps:** Compare `root_cause_step` as a raw 1-indexed
   integer. Do not try to align steps by description similarity across annotators.
   Because plan lengths vary per trace, kappa is computed on the raw step index values.
   This is technically a variable-range ordinal label set — see IAA Metric Notes below.

2. **Excluded traces:** If either annotator marked a trace as excluded, drop it from the
   IAA computation. Exclusions signal data quality problems, not annotation disagreement.

3. **Free-text reasoning:** `root_cause_reasoning` is shown side-by-side in the
   per-trace table for manual inspection. No automatic scoring.

4. **Timing:** No explicit performance constraint. For N ≤ 500 traces, pure Python +
   pandas is sufficient. No optimization needed.

5. **Collaborator handoff:** Canonical method is a git PR to `data/annotations/collaborator/`.
   Alternative: email or share the folder directly and you merge manually. Document both
   in the README with the PR method as the default.

6. **Paper-ready standard:** Report rounds all statistics to 2 decimal places, includes
   Landis-Koch kappa interpretation scale (e.g., "0.61–0.80 = substantial agreement"),
   and provides a methods note suitable for copy-paste into a paper.

## IAA Metric Notes

`root_cause_step` is an ordinal integer with trace-varying max (1 to len(plan)). Cohen's
kappa treats labels as unordered categories, so two annotators both choosing "step 2 of 5"
vs "step 2 of 8" are treated identically. This is the correct behavior — the step index
is the label regardless of plan length. However, report includes:

- **Percent exact agreement** (robust, no label-set assumption).
- **Cohen's kappa** (sklearn, unweighted — treats each step index as a distinct category).
- **Mean absolute step distance** |step_A - step_B| — more interpretable than kappa
  for ordinal data with variable range.
- **Confusion matrix** capped at step indices 1–8 (covers >95% of typical plans); higher
  indices bucketed into "8+". This prevents a sparse, unreadable matrix.

No 95% CI on kappa — `sklearn.metrics.cohen_kappa_score` does not return CI and
bootstrapping adds complexity without proportional value for typical N in this study.
Report notes this limitation.

**Guard clause:** If fewer than 2 traces overlap between the two annotators' completed
sets, `compute_iaa.py` prints a clear error and exits:
```
Error: only N overlapping traces found. Need at least 2 to compute agreement.
```

## Open Questions

1. **Minimum N for kappa stability:** Cohen's kappa on small N (< 20) is unstable. If
   fewer than 20 traces overlap, the report will note this prominently. How many traces
   does each annotator plan to complete? (Answer determines whether to note this caveat.)
2. **Should step_annotations status values be standardized?** Currently free-form.
   If annotators use different status strings (e.g., "failed" vs "error"), per-step
   categorical agreement will undercount matches. Recommend agreeing on a fixed vocabulary
   before annotation begins: ["completed", "failed", "partial", "skipped", "unknown"].

## Success Criteria

- Collaborator can set up and annotate without asking questions.
- `python compute_iaa.py` runs in < 5 seconds and produces `iaa_report.md`.
- Report includes Cohen's kappa, confusion matrix, and per-trace flags.
- High-disagreement traces are surfaced and include both annotators' reasoning for easy review.
